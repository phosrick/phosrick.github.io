# 我以为慢只是因为没有并发

在当前代码里：因果边任务的核心结构可以简化为：

```python
node_batches = batch_list(node_names, batch_size)
k = len(node_batches)

batch_pairs = list(product(range(k), repeat=2))

with ThreadPoolExecutor(max_workers=max_workers) as executor:
    futures = {
        executor.submit(
            causal_agent.extract,
            node_batches[i],
            node_batches[j],
        ): (i, j)
        for i, j in batch_pairs
    }
```

写关系抽取流水线时，我最初的判断很直接：节点已经抽出来了，接下来就是让大模型判断哪些节点之间有关系。API 调用慢，就分批，再用线程池并发就能提效。

从调度机制上看，线程可以重叠多个 API 请求的等待时间；日志也开始同时滚动，不再是一批结束后才轮到下一批。这种“大家都在同时干活”的画面，还是很容易让我产生一种错觉：性能问题已经解决了。

后来我重新数了一遍提交给线程池的 Future，才意识到我只是让同一批工作更早挤进了队列，并没有减少工作本身。

并发改的是等待方式，不是任务数量。任务数仍然按平方增长，只是被同时滚动的进度条盖住了。

# 批次数 k → k² 次 LLM 调用，这是不是 bug

V5 的 pre 阶段先从参考材料中抽取节点，再生成三类关系：

- 因果关系：一个节点是否影响另一个节点；
- 信息支持关系：某条信息是否为某个决策提供依据；
- 证据更新关系：已知信息是否改变了对未知信息的判断。

单个节点本身无法说明关系。要判断 `A -> B`，模型至少需要同时看到 A 和 B。最简单粗暴的写法自然是把左侧集合和右侧集合全部组合：

```text
列表 A 的每一批 × 列表 B 的每一批
```

1. 单次 Prompt 放不下无限多节点，必须分批；
2. 不希望在抽取前漏掉关系，于是让模型看完所有可能的批次组合。

麻烦恰恰来自第二点。为了不漏掉跨批次关系，当前的方案采用了最昂贵的穷举方案，让 LLM 检查所有批次之间可能存在的有向关系，尽量避免漏边。

这不是 bug，这是一个以召回率优先的算法设计。

# 第一个坑：批处理控制了上下文，却没有消除平方项

product(range(n_batches), repeat=2) 表示对批次编号做两次笛卡尔积，也就是生成所有有序二元组，假设 n_batches = 3，product(range(3), repeat=2) 结果是：

```text
[
    (0, 0), (0, 1), (0, 2),
    (1, 0), (1, 1), (1, 2),
    (2, 0), (2, 1), (2, 2),
]
```

用当前默认 `B = 50` 做纯理论计算：

| 节点数 N | 批次数 k | 因果边调用数 k² |
| -------: | -------: | ---------------: |
|       10 |        1 |                1 |
|       50 |        1 |                1 |
|       51 |        2 |                4 |
|      100 |        2 |                4 |
|      250 |        5 |               25 |
|      500 |       10 |              100 |
|     1000 |       20 |              400 |

这张表是按当前代码计算出来的任务数量。它揭示了两个容易被日志速度掩盖的事实。

第一，规模增长不是平滑的。50 到 51 个节点，批次数从 1 变成 2，调用数会从 1 变成 4。系统的耗时可能在批次边界出现台阶，而不是随着节点数均匀增加。

第二，调大批次虽然能减少调用数，却会把更多节点塞进单次 Prompt。调用变少不等于推理一定更快：输入 Token 增加，模型要检查的批内节点对更多，输出也更容易变长或被截断。

这之后我才把“批大小”看成一个多目标参数。它同时影响：

- 请求数量；
- 单次输入长度；
- 批内候选对数量；
- 输出长度和 JSON 稳定性；
- 单次失败的影响范围；
- API 的 RPM 与 TPM 消耗方式。

所以它不是越大越好，也不是越小越安全。

# 第二个坑：我以为 `max_workers=5` 就是最多 5 个请求

关系抽取不是一个线程池，而是嵌套线程池。

外层先把三类关系并发起来：

```python
with ThreadPoolExecutor(max_workers=3) as executor:
    future_causal = executor.submit(extract_causal_concurrent)
    future_info = executor.submit(extract_info_concurrent)
    future_update = executor.submit(extract_update_concurrent)
```

每个分支内部又创建自己的线程池：

```python
with ThreadPoolExecutor(max_workers=self.max_workers) as executor:
    # 提交该关系类型的全部批次对
```

当前 `PipelineRunner.max_workers` 默认是 5。三类关系都有任务时，理论上在途 LLM 请求上限不再是 5，而可能接近：

```text
3 × 5 = 15
```

外层的 3 个线程主要负责启动并等待内层线程池，占用 API 配额的是内层请求。问题在于，`max_workers` 这个名字看起来像全局并发配置，实际上只约束单个关系分支。

这类配置最容易让我误判系统资源：我以为自己把并发调到了 5，供应商看到的却可能是三个分支一起冲上来。

而且，线程池限制的是“同时执行多少个任务”，不是“总共创建多少个任务”。代码会先构造完整的 `batch_pairs`，再为所有组合创建 Future。即使只有 5 个 worker，其余任务也已经进入执行器队列，不能形成上游背压。

当批次数还小时，这种写法清楚、直接，也完全能用。规模继续增大后，它会带来三个问题：

1. 队列和 Future 会提前占用内存；
2. 任务一旦全部提交，就不容易根据实时限流或错误率暂停生产；
3. 三个分支各管各的线程池，没有共同的 RPM、TPM 和在途请求预算。

# 并发是调度，不是减法

知道了嵌套线程池会把在途请求乘到 15，直觉反应是把 `max_workers` 调小。但这只解决了”并发失控”，没解决”任务过多”——两个问题恰好共用一个参数，容易被当成同一件事。

我一开始把”速度”等同于墙钟时间：启动到落盘，越短越好，但对 LLM 流水线来说太窄了。速度至少是四个维度的乘积：

| 维度                                         | 受`max_workers` 影响？ |
| -------------------------------------------- | ------------------------ |
| 调用量 — 一共发出了多少次请求               | 否                       |
| Token — 总输入和总输出是多少                | 否                       |
| 吞吐 — 单位时间完成多少个有效批次           | 是                       |
| 失败面 — 有多少请求可能超时、限流或解析失败 | 是，反向                 |

全笛卡尔积的 k² 个任务，不会因为 worker 从 5 变成 20 就自动缩成 k 个。

而且，吞吐和失败面往往是跷跷板。供应商限制 RPM，并发过高就触发限流；限制 TPM，大批次 Prompt 先耗尽 Token 窗口；尾延迟一长，最后几个慢请求仍然卡住整个阶段。把 worker 调大，只是把问题从”排队排太久”换成了”大家一起撞墙”。

并发在这里是调度策略，不是复杂度优化。它重新分配了等待发生的位置，但没有减少等待的总量。

# 第三个坑：分支隔离让流程更能跑完，也让失败更容易被低估

当前每个 Future 都单独捕获异常：

```python
for future in as_completed(futures):
    try:
        arcs = future.result()
        all_arcs.extend(arcs)
    except Exception as exc:
        logger.error("batch failed: %s", exc)
```

外层三类关系也分别捕获异常，失败分支保留空列表，其他分支继续完成。

这种 best-effort 行为有实际价值：一个批次超时，不至于让已经完成的几十个批次全部作废。但它也带来一个性能治理之外的问题：如果最终只看到关系数量变少，我很难判断是文本中确实没有关系，还是某几个批次技术失败了。

并发越高，部分失败越可能同时出现。没有批次级状态、重试次数和完成比例时，“跑完了”不等于“跑完整了”。

这篇不展开错误语义设计，但它提醒我：性能优化不能只记录耗时。至少还应该同时保存：

- 计划任务数；
- 成功、空结果和失败任务数；
- 每个分支的完成比例；
- 重试次数与错误类型；
- 最终结果是否允许进入下游。

# 如何改进

## 剪枝

要减少工作量，得在 LLM 前面剪枝：把候选批次对送给 LLM 之前，先砍掉那些不可能或不太可能有关系的组合，只让模型判断真正值得判断的那一小部分。

具体到三种剪枝手段：

### 1. 先用确定性规则排除不可能的类型组合

信息支持边已经只比较“信息节点 × 决策节点”，证据更新边也只比较“已知信息 × 未知信息”。这正是有效剪枝：在不调用模型之前，先用 Schema 缩小候选空间。

因果边当前使用全部节点。下一步可以根据业务允许的节点类型、禁止的自环、已知层级或方向规则继续筛选。但这些规则必须来自明确的图 Schema，不能为了省调用随意删候选。

### 2. 用便宜的 Retriever 生成 Top-K 邻居

可以先用 BM25、Embedding 或混合召回，为每个节点找最可能相关的 K 个节点，再让 LLM 判断这些候选关系。

全量比较的候选规模近似是：

```text
O(N²)
```

如果每个节点只保留 K 个候选邻居，K 远小于 N，候选规模可以降到：

```text
O(NK)
```

代价是引入召回上限：Retriever 没找回来的关系，后面的 LLM 永远看不到。因此优化不能只比较耗时，还要比较边召回率。

### 3. 先粗筛，再精判

另一种做法是两阶段关系抽取：

```text
规则/小模型/低成本模型：判断哪些批次值得展开
                    ↓
高能力模型：抽取方向、关系类型和不确定性
```

这样昂贵模型不再承担全空间搜索，只负责难判断的候选。它本质上是 Generator-Verifier 思路在性能方向上的一次延伸。

### 4. 缓存参考图，而不是反复重建

V5 已经把参考图放在 pre 阶段，使同一道题的多位参与者共享节点和关系，这是现有设计里非常重要的成本摊薄。

还可以继续补输入哈希、模型版本、Prompt 版本和参数哈希。只有这些都一致时才复用关系抽取缓存，避免“文件还在，所以默认有效”的隐式判断。

## 背压：给流水线装一个总阀门

剪枝管”送多少”，背压管”送多快”。前者减少候选组合，后者控制剩余任务以什么速度进入 Provider。剪枝再彻底，如果剩下的请求仍然以 3 × max_workers 的并发量无限制地冲进去，Provider 照样限流。

当前的问题不是没有限流，而是限流藏在每个分支自己的 `max_workers` 里，三个分支各管各的，没有人在看全局。直观地看：

```text
因果分支 → 自己的线程池 → 提交 k² 个任务
信息分支 → 自己的线程池 → 提交 m×n 个任务
更新分支 → 自己的线程池 → 提交 p×q 个任务
        ↓
    三个分支同时往 Provider 冲
```

每个分支的线程池只知道”我最多同时跑 5 个请求”，不知道另外两个分支也在跑。三个一合起来，Provider 可能同时收到 15 个请求。而且所有任务是一次性全部塞进队列的——生产者完全不关心消费者能不能消化，这就是没有背压。

有背压的结构是所有 LLM 调用共享一个运行时预算：

```text
                    ┌─ 因果任务
producer / queue ───┼─ 信息任务 ──► 全局请求预算 ──► Provider
                    └─ 更新任务
```

跟现在的区别不是多了一个框，而是多了三层控制：

**第一层，并发上限。** 全局 semaphore 限制同时在途的请求数，不管是哪个分支发起的。这解决了 `max_workers` 只约束单分支、三个一合起来就翻倍的问题。

**第二层，速率上限。** Semaphore 只管”同时有多少”，不管”每分钟有多少”。RPM 和 TPM 需要令牌桶——以固定速率发放请求额度，用完了就等，而不是攒到每分钟开头一起冲。

**第三层，异常恢复。** 即使前两层都设好了，仍然可能触发 429。关键是不能让所有线程同时立刻重试——那等于把一次限流放大成 retry storm。需要指数退避加随机抖动，让重试分散到不同的时间点。

三层缺一不可。只有 semaphore 没有令牌桶，请求会集中爆发；只有令牌桶没有退避，一次 429 可能连锁失败。

## 性能实验：怎么才算”真的变快”

动代码之前，我会先固定一套公开或脱敏样本，记录以下数据：

| 指标                     | 为什么要记录               |
| ------------------------ | -------------------------- |
| 节点数与各类型节点数     | 解释三个候选空间的规模     |
| 批大小与计划任务数       | 验证公式与实际调度一致     |
| 每类边的成功/空/失败批次 | 防止用残缺结果换速度       |
| 输入/输出 Token          | 计算真实成本，不只看调用数 |
| P50/P95 单请求延迟       | 识别尾延迟和批大小影响     |
| 阶段墙钟时间             | 衡量用户实际等待           |
| 最大在途请求数           | 验证背压是否生效           |
| 边 Precision/Recall      | 确认剪枝没有破坏质量       |

然后比较三组方案：

```text
A. 当前全批次笛卡尔积
B. 类型与确定性规则剪枝
C. 规则剪枝 + Retriever Top-K + LLM 验证
```

每组都用相同节点和人工标注边，分别改变 batch size、Top-K 和全局并发。最后看的不是某一个最快配置，而是延迟、Token、错误率和边召回率之间的 Pareto 前沿。

这才是我认为“并发真的变快”的完整证据。只看进度条滚得更快远远不够。

# 复盘心得

以前看到慢，我第一反应是加并发。现在我会先数任务——沿着调用链看清哪些决策在放大工作量，`max_workers` 约束的到底是哪一层，Provider 那边又在发生什么。墙钟时间之外，调用量、Token、失败面、结果完整性，这些才是真正的账单。

Multi-Agent 流水线危险的地方，往往不是某个 Agent 特别慢，而是多个看起来合理的局部决策叠在一起。单独看都没问题，合起来却把任务数、并发和失败面一起放大了。

所以加了线程池还是慢，不是线程池的错。是判断顺序的问题。正确的顺序应当是：

```text
先计算候选空间
再减少工作总量
然后设置全局预算
最后才调 worker 数
```
