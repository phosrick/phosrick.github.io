# 先从圆桌讨论的论题开始

如果让大模型读完一场圆桌讨论，然后直接回答"谁的表现最好"，它当然可以给出一段听起来很完整的评语。

但真正交付一份候选人评估报告时，我还需要回答三个问题：

```text
这个结论依据了哪段发言？
这段发言对应题目中的哪个事实、决策或目标？
同一个标准能否用来比较这一组人的表现？
```

只把整段讨论拼进 Prompt，通常很难回答这三个问题。大模型可能记住了观点，却没有稳定地区分"事实""行动"和"目标"；它也可能因为某个人发言更长，就把表达量误当成能力。

V5 的做法是先把参考材料编译成一张影响图，再把每位参与者的发言映射到这张图上。图上的节点和关系成为中间证据，之后才进入规则评分、图算法和 LLM 摘要。

这让我重新理解了 RAG：检索增强不一定是给聊天机器人找几段文本，也可以是把开放发言收敛成下游能够复算的结构化证据。

# 先把 RAG 说清楚

RAG 的全称是 Retrieval-Augmented Generation，最常见的流程是：

```text
文档 -> 切块 -> Embedding -> 向量索引
问题 -> 相似度检索 -> Top-K 文本 -> LLM 回答
```

这里的 Top-K 是"按相关性排序后只保留前 K 条"。它控制上下文长度、成本和噪声，但不是 RAG 的定义本身。

对 V5 来说，检索到的上下文不只是用来生成回答，还要喂给确定性的图模式和规则计算。基于这一点，我认为：

> V5 是一条结构化检索增强评估链。它具备 RAG 的关键特征，但不是典型向量 RAG，也不是完整的 GraphRAG。

它既说明了外部知识、Query、Retriever 和 Retrieved Context（检索上下文）的确存在，也没有把尚未实现的 ANN、社区检索和混合召回包装成现成功能。

# 案例：从"下午人更多"到两个节点

先看普通向量 RAG 会怎么处理。假设系统里存着很多文本段落：

段落 1：下午客流比上午高。 \
段落 2：高峰期应该增加员工。 \
段落 3：可以调整营业时间。 \
段落 4：线上预约可以减少排队。 \
段落 5：应该降低办公成本。

用户问：下午人多的时候应该怎么办？

## 1. 普通向量 RAG 在找什么？

普通向量 RAG 会计算问题和这些段落的相似度，然后返回最相关的几段：

Top-1：下午客流比上午高 \
Top-2：高峰期应该增加员工 \
Top-3：可以调整营业时间 \
...

它回答的是：

> 哪几段原文和这个问题最相似？

这是 Top-K 文本检索。

## 2. V5 在找什么？

V5 走的是另一条路。它事先从题目的参考材料中抽取好节点——不是上面那 5 段语料，而是出题方提供的一份参考答案级别的材料。可以把这些节点想成题目提前准备好的三张概念卡片：一张描述事实，一张描述行动，一张描述目标：

[信息] 下午到店客流高于上午 \
[决策] 调整网点服务时段 \
[效用] 提升网点经营效率

参与者说：下午人明显更多，可以把更多员工排到那个时段，别让上午和下午用同一套安排。

这时 V5 不是去大量资料里找相似原文，而是让 LLM 在一组有限的参考节点中做语义判断：

> 这段话对应上面整理好的哪些节点？

判断结果是：

```json
{
  "matched_nodes": [
    "下午到店客流高于上午",
    "调整网点服务时段"
  ]
}
```

这段 JSON 的意思不是说它找到了两段最相似的文本，而是：

这位参与者提到了"客流增加"这个信息，
也提到了"调整服务时段"这个决策。

节点命中之后，系统还可以从参考图中找出头尾都已命中的候选边，再回到参与者原文中验证这条关系是否真的被表达出来。

参考图中有两条边：

```text
下午到店客流高于上午 -> 调整网点服务时段
调整网点服务时段 -> 提升网点经营效率
```

到这里，一条最小的检索增强链就出现了：

```text
参与者发言
  -> 命中参考节点
  -> 诱导候选关系
  -> 回原文验证关系
  -> 结构化证据
  -> 评分与摘要
```

# 影响图究竟存了什么

如果只把参与者发言和参考答案做向量相似度，最多能回答"它们像不像"。评分器还需要知道表达在题目中扮演什么角色。

## 1. 影响图的基本形式

它可以形式化成一个有方向、带关系类型的图：

```text
G = (V, E)
```

其中：

- `V` 是节点集合；
- `E` 是边集合；
- 节点表示题目中的信息、决策和目标；
- 边表示节点之间的因果、支持或证据更新关系。

## 2. 节点：事实、行动和目标

当前 V5 主要把参考材料中的概念分成三类。

### 信息节点

表示事实、数据、现象、证据或观察：

```text
下午到店客流高于上午
顾客等待时间变长
员工在高峰期不足
```

它们回答的是："现在发生了什么？我们知道什么？"

### 决策节点

表示行动、方案、判断或资源配置：

```text
调整网点服务时段
增加下午班次
优化线上预约流程
```

它们回答的是："可以采取什么行动？"

### 效用节点

表示目标、结果或希望改善的方向：

```text
提升网点经营效率
降低顾客等待时间
提高客户满意度
```

它们回答的是："采取行动之后，希望达到什么结果？"

所以，影响图不是把三段文字放在一起，而是给每段表达分配一个角色：

```text
"下午更忙"是信息
"调整排班"是决策
"提高效率"是目标
```

案例中的参考材料可以整理成：

```text
节点 A：下午到店客流高于上午
节点 B：调整网点服务时段
节点 C：提升网点经营效率
```

## 3. 边：支持、影响和证据更新

当前 V5 保存三类主要关系。

### 因果关系

表示一个节点可能导致或影响另一个节点：

```text
增加下午员工 -> 缩短高峰期等待时间
```

### 信息支持关系

表示一个信息为某个决策提供依据：

```text
下午客流高于上午 -> 支持调整服务时段
```

### 证据更新关系

表示新证据会改变人们对另一个判断的相信程度：

```text
连续三天下午客流增加 -> 提高"下午高峰会持续"的可信度
```

把案例画出来，就是：

```text
[信息] 下午到店客流高于上午
          |
          | 信息支持
          v
[决策] 调整网点服务时段
          |
          | 因果影响
          v
[效用] 提升网点经营效率
```

代码中的三类关系分别保存为 `caus_arcs.json`、`info_arcs.json` 和 `update_arcs.json`。每条边至少有起点 `head` 和终点 `tail`，还可能记录正向、负向、确定或不确定等条件信息。

## 4. 参考材料如何变成这张图

影响图由 `pre` 阶段建立，不是参与者讨论之后临时拼出来的：

```text
参考材料
  -> 文本润色
  -> 分块
  -> LLM 抽取节点
  -> 节点质量检查
  -> 节点分类
  -> LLM 抽取关系
  -> 保存参考图
```

主要产物包括：

```text
node_refanswer.json   # 参考节点
caus_arcs.json        # 因果关系
info_arcs.json        # 信息支持关系
update_arcs.json      # 证据更新关系
```

这些文件共同定义了本题的有限证据空间：哪些事实可能被讨论，哪些行动和目标可能被连接。同一道题的多个参与者共享这张参考图，避免每个人各用一套漂移的评分标准。

## 5. 从节点命中到关系验证

节点命中只是第一步。系统随后从参考图中筛选头尾都已命中的候选边，再回到参与者原文验证这条关系是否真的被表达——这条链路在前面案例里已经走通过一次。

这一步真正改变的是评分器拿到的证据形态：开放的自然语言被收敛成结构化证据，可以复算、可以回溯。

这张图带来两个同时存在的效果：

1. **可比性提高**：A 说"下午增派人手"，B 说"重新安排高峰班次"，都可以映射到同一个决策节点。
2. **开放性下降**：参考图中没有的好观点，不会在节点召回阶段凭空出现。

所以影响图是一个有类型、有方向、带白名单约束的知识表示。它让评分更容易复算，也把上游抽取遗漏变成了召回上限。

# Retriever 没有计算向量，而是在白名单里做判断

当前节点检索的核心代码在 `src/pipeline/runner.py`。参考节点每 20 个分成一批，每位参与者和每一批节点组成一个任务：

```python
node_batch_size = 20
ref_node_batches = batch_list(ref_names, node_batch_size)

all_tasks = [
    (sid, batch_idx, speaker_texts[sid], ref_node_batches[batch_idx])
    for sid in speaker_texts
    for batch_idx in range(len(ref_node_batches))
]
```

每个任务要求 LLM 只返回命中的节点编号：

```python
prompt = f"""
请判断以下概念是否在给定的文本中被提到或讨论过。

【文本内容】
{text[:3000]}

【待判断的概念列表】
{nodes_str}

只输出被提到的概念编号，用逗号分隔；都没有则输出"无"。
"""
```

这是一种受限语义判断：候选集合已经由参考图给出，LLM 负责处理同义改写、隐含表达和口语差异，返回结果直接落到评分器认识的节点 ID 上。

若有 `S` 位参与者、`N` 个参考节点，批大小为 20，节点判断任务量大致是：

```text
S × ceil(N / 20)
```

当前没有相似度分数、Top-K、reranker 或 ANN 索引。节点越多、参与者越多，LLM 判断任务就越多。它适合题目级、有限节点集合；如果参考语料扩大到跨题目、跨行业的长期知识库，就应该考虑 BM25、Embedding 或混合召回。

# 关系检索为什么要"先筛，再验"

节点命中还不够。逻辑推理、目标拆解和影响预判关心的是：参与者有没有把信息、行动和目标连接起来。

V5 没有让 LLM 自由生成所有关系，而是先用命中节点从参考图中诱导候选边：

```python
candidate_arcs = [
    arc for arc in all_ref_arcs
    if arc["head"] in node_names
    and arc["tail"] in node_names
]

if not candidate_arcs:
    return []

verified_arcs = self.verify_agent.verify(text, candidate_arcs)
```

`ArcVerifyAgent` 再回到参与者文本，逐条判断候选关系是否真的存在，还会判断关系是正向、负向、确定还是不确定。

这个顺序的价值是把关系生成限制在参考图允许的空间内：

```text
节点召回
  -> 参考图筛选候选边
  -> 原文验证
  -> 个人关系证据
```

# 检索证据到底增强了什么

传统 RAG 往往把 Top-K 文本拼进上下文，再生成回答。V5 的检索上下文有三类消费者：

1. **确定性或半确定性评分**：读取命中节点的类型、数量和图模式，计算逻辑思维、目标拆解、信息收集等维度；
2. **LLM 判断**：验证候选关系、评估方案可行性等语义任务；
3. **LLM 生成**：生成维度摘要、能力总结和最终报告。

# 什么时候引入向量数据库

当前方案和引入向量数据库两种方案的适用边界不同：

| 维度 | 当前 LLM 白名单判断 | Embedding / BM25 Top-K |
|---|---|---|
| 候选空间 | 单题目的有限参考节点 | 大规模跨题目文本或节点 |
| 结果形式 | 命中 / 未命中节点 ID | 排序后的候选及相似度 |
| 结构约束 | 天然保留节点类型和关系白名单 | 需要后处理补充图约束 |
| 召回成本 | 随参与者数和节点批次线性增加 | 索引后查询更快 |
| 开放域发现 | 弱，图外观点容易漏掉 | 更适合发现未预先列出的相似内容 |
| 可解释性 | 可直接回到参考节点 | 需要保存文本块、分数和来源 |

当前题目级参考图是有限的，同一张图服务一批参与者。直接引入数据库会同时带来 Embedding 版本、索引同步、元数据过滤、租户隔离、重建和可观测性问题，引入向量数据库并不能证明检索效果会变得更好。

> 当参考空间变成跨题目、大规模、开放域，或者在线并发和延迟成为真实瓶颈时，再升级存储和服务形态。

# 可改进的地方

## 当前的局限性

### 1. LLM 需要扫描全部参考节点

当前每 20 个参考节点组成一批，每位参与者都要和所有批次分别调用 LLM。任务量随参与者数和节点数线性增长，前面算过是 `S × ceil(N / 20)`。

题目级节点集合较小时，这个方案可以工作；当知识空间扩大到跨题目、跨岗位甚至跨行业时，调用数、Token 和延迟都会随节点规模上升。

### 2. 只有"命中 / 未命中"，没有排序和置信度

当前节点结果主要是一个集合：哪些节点命中了，哪些没有命中。系统没有相似度排序，也没有召回置信度，因此很难观察"这个节点是非常相关，还是勉强相关"，也不容易诊断空召回究竟是表达偏离、参考图遗漏还是模型判断异常。

### 3. 参考图白名单会漏掉图外的新观点

参考图提高了不同参与者之间的可比性，也把召回范围限制在已有节点中。如果参与者提出参考答案中没有的新方案，当前节点匹配不会自然地为它创建新的结构化证据。

### 4. 业务空结果和技术失败可能长得一样

API 凭据缺失、调用异常或解析失败时，部分路径会返回 `[]`。下游难以区分：

```text
确实没有命中任何节点
Retriever 没有正常工作
```

如果空列表继续进入评分，技术失败就可能被误解为候选人没有相关表现。

### 5. 召回效果还没有正式基线

当前尚未建立节点和关系的人工 golden set，也没有系统比较关键词、BM25、Embedding Top-K 和 LLM 全量判断。因此，现在还不能用数据证明哪种 Retriever 在准确率、成本和延迟之间更合适。

## 引入向量索引可以改进什么

### 1. 让 LLM 从"扫描全部节点"变成"判断少量候选"

向量索引可以先从全部节点中召回 Top-M，再让 LLM 做语义判定：

```text
全部参考节点
  -> BM25 / Embedding Top-M
  -> LLM rerank 或语义判定
  -> 最终命中节点
```

假设有 8 位参与者、2000 个参考节点，当前全量分批判断需要：

```text
8 × ceil(2000 / 20) = 800 个判断任务
```

如果向量索引先为每位参与者预筛 50 个候选节点，后续 LLM 判断任务大致变成：

```text
8 × ceil(50 / 20) = 24 个判断任务
```

### 2. 降低重复 Token、延迟和 API 压力

当前同一份参与者文本会随着节点批次重复进入 Prompt。向量索引建立后，参考节点只需在 `pre` 阶段计算一次 Embedding，同一道题的参与者可以复用索引。LLM 只接收预筛后的候选节点，重复输入和并发请求都会减少。

### 3. 支持更大的跨题目知识空间

当系统积累多道题目、多个岗位和多个行业的参考节点时，可以先按元数据过滤：

```text
岗位
行业
题目类型
节点类型
图版本
```

再执行向量检索。这样系统才有可能从单题目的离线评估，演进为可以复用跨题目知识的评估平台。

### 4. 将召回细化到话轮或滑动窗口

有了快速索引后，可以不再把整位参与者的文本当作唯一 Query，而是按话轮或滑动窗口召回：

```text
每个话轮
  -> 召回候选节点
  -> 保存 turn_id、timestamp 和原文片段
  -> 按参与者聚合
```

这能让最终报告更容易回答"这个判断来自哪一句话"。

### 5. 获得排序分数和可评测的召回过程

向量检索可以返回候选节点及相似度：

```text
0.91  调整网点服务时段
0.86  增加高峰期员工
0.73  优化预约流程
```

这些分数不能直接当作能力分数，但可以用于选择候选范围、观察异常、比较 Embedding 模型，并计算 Recall@M、MRR、nDCG、空召回率和平均延迟等指标。

### 6. 用混合检索覆盖不同表达

Embedding 擅长同义改写，BM25 擅长精确数字、专有名词和业务术语。更合理的候选生成方式是：

```text
BM25 召回
       \
        -> 候选合并 -> LLM 语义判定
       /
Embedding 召回
```

例如"下午 3 点到 5 点客流增加 37%"同时包含精确数字和"下午高峰客流增长"的语义。两种召回合并，比单独依赖关键词或向量更稳。

### 7. 为图外新观点保留一条发现通道

向量索引可以从更大的知识空间中找出与新方案相近的概念，再把结果分成两类：

```text
参考图内节点 -> 用于统一、可比较的结构化评分
参考图外候选 -> 用于创造性分析和人工复核
```

这样既保留参考图的评分标准，也不会让白名单完全压制参考答案之外的有效观点。

# 向量索引不能解决什么

向量索引解决的是"从哪里快速找候选"，不能解决下面的问题：

- 参考图自身漏节点、错分类或关系方向错误；
- 参与者是否真的表达了某条因果或支持关系；
- 个人验证边尚未统一进入评分的数据流缺口；
- 能力分数是否与专家判断和后续表现一致；
- 音频、转写、语言风格和模型带来的公平性偏差。

所以完整职责仍应保持分层：

```text
BM25 / Embedding：从大空间找候选
LLM：判断候选是否真正对应参与者表达
影响图：保留节点类型、关系方向和题目约束
原文验证：确认关系是否真实存在
评分器：把结构化证据转换成能力指标
```

每一层解决的是不同的问题，换掉其中任何一层都要重新回答"这一层的职责谁来接"。
