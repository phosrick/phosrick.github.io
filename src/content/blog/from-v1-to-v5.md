# 一、背景概述

这个项目开始于 2024 年 5 月 22 日。

它要解决的问题一直很具体：输入一场圆桌讨论的转写稿，分析每位参与者在讨论中的行为和贡献，最后在六大能力、十六个子维度上给出评分和评价，用于候选人的综合能力评估。

现在，项目已经到了第五版。

这里的“第五版”不是改了五轮 Prompt，也不是给同一个服务换了五次模型。过去几次迭代基本都伴随着大范围重构，甚至是推翻前面的工程结构重新开始。

由于 V1 到 V3 没有代码备份，因此我无法写出一条从 V1 到 V5 每一步都清晰可见的演进曲线。我要复盘的是目前真正有证据的部分：**V4 已经具备哪些能力，V5 为什么仍然值得整体重写，重写改善了什么，又可能损失了什么。**

# 二、关于 V4 

在讨论 V5 之前，V4 已经具备相当完整的业务能力：

- 从参考语料中抽取影响图节点；
- 对节点进行分类；
- 抽取因果、信息支持和证据更新三类关系；
- 从小组讨论中拆分个人语料；
- 为参与者匹配节点和关系；
- 准备推理模式数据；
- 计算十六个能力子维度；
- 使用 LoRA 模型抽取节点；
- 调用 DeepSeek、智谱等外部模型；
- 使用 Hawkes 点过程计算思维激发；
- 使用 pgmpy 进行图上的 d-connection 判断；
- 并发执行节点、关系和评分任务；
- 生成摘要、图表和最终报告；
- 通过子进程按顺序编排全流程；
- 保存 PID，并在单脚本失败后进行一次重试。

也就是说，V5 并不是从零开始补齐业务能力。很多核心算法和业务定义在 V4 中已经存在。

V4 真正的问题主要在工程组织方式。

## 1. 文件名就是执行顺序

V4 的主目录大致是这样的：

```text
1.1.1 get_node_refanswer.py
1.1.2 node_classification.py
1.2 get_arcs_refanswer.py
1.3 json2csv.py
1.4 get_embeddings.py
2.0 split_corpus.py
2.1 reply_preprocess.py
2.2 get_nodes_reply.py
2.3 get_arcs_reply.py
3 data_processing.py
4.1 get_arc.py
4.2 arc_drop_dup.py
5 reasoning_pattern_prepare.py
6.1.1 思维能力-逻辑思维能力.py
...
6.6.3 讨论控场能力-收敛性.py
7 summary.py
generate_report_detail.py
```

脚本编号同时承担了三个职责：执行顺序、业务分类和模块说明。

非常直接：打开目录就能看见流水线先跑什么、后跑什么，不需要理解框架。

但随着脚本增多，文件系统本身开始变成一个脆弱的工作流引擎：重排阶段要改编号，复用能力要跨文件复制代码，公共协议只能靠开发者记忆。

## 2. 配置对象知道所有细节

V4 的 `run_pipeline.py` 先构造一个很大的配置字典，里面包含每一个阶段的输入路径、输出路径、模型路径、十六个维度的分数目录、摘要目录和绘图目录。

随后，编排器按照一个 `SCRIPTS` 列表依次启动三十多个 Python 文件：

```python
for script_name in SCRIPTS:
    result = subprocess.run(
        [sys.executable, script_path, "--config", config_json],
        check=True,
        capture_output=True,
        text=True,
    )
```

这个设计有很强的进程隔离，单个脚本也容易手工运行，但代价是：

- 中间协议散落在文件路径和 JSON 字段中；
- 修改目录结构会影响大量脚本；
- 很难在进程内共享模型、缓存和已加载数据；
- 错误信息需要跨子进程拼接；
- 评分器之间大量重复初始化和文件读取。

## 3. 一个维度就是一个完整程序

V4 的十六个维度分别对应独立脚本。每个脚本通常自行处理：

- 参数解析；
- 配置加载；
- 数据读取；
- 分数计算；
- 团队平均值；
- LLM 摘要；
- 并发；
- 文件输出；
- 可选绘图。

这保证了每个维度能够独立运行，却也让相同的生命周期代码重复出现在多个文件中。

# 三、关于 V5

V5 的目录结构变成：

```text
src/
├── pipeline/
│   ├── node_agents.py
│   ├── arc_agents.py
│   ├── node_extraction.py
│   └── runner.py
├── scoring/
│   ├── base.py
│   ├── data_manager.py
│   ├── graph_manager.py
│   ├── embedding_manager.py
│   ├── runner.py
│   └── scorers/
├── reporting/
├── finetune/
└── utils/
```

这个结构真正改变的是代码所有权。

## 1. 从“步骤脚本”变成“领域模块”

V4 用文件编号表达流程；V5 用包表达职责：

- `pipeline` 负责文本到节点和关系；
- `scoring` 负责十六维计算；
- `reporting` 负责总结、图表和报告；
- `finetune` 负责本地模型训练和比较；
- `utils` 负责数据库、文本、通知和通用模型调用。

执行顺序不再写进文件名，而是由 Runner 明确编排。

## 2. 从独立评分程序变成 Registry

V5 为评分器增加 `BaseScorer`，并使用 `SCORERS` 字典注册十六个维度：

```python
SCORERS = {
    "1_1": LogicalReasoningScorer,
    "1_2": AbstractionScorer,
    # ...
    "6_3": ConvergenceScorer,
}
```

评分器共享统一的初始化方式、结果结构、摘要阶段和绘图阶段。运行器可以选择部分评分器，也可以并发执行全部评分器。

这让“增加一个能力维度”从“复制一个完整程序并加入脚本列表”，变成“实现一个评分类并注册”。

当前 Registry 仍不完美。部分复杂评分器覆盖了整个 `run()`，而单人计算方法只是占位。这说明统一接口还没有彻底覆盖所有算法，但扩展边界已经比 V4 清晰。

## 3. 从重复读盘变成共享数据管理

V5 引入 `DataManager`，通过惰性属性缓存参考节点、节点类型、边、参与者数据和派生索引。

十六个评分器不再各自理解所有目录和文件格式，而是依赖一个共享的数据访问层。

这降低了重复 I/O，也把路径兼容和输入格式变化集中到一个位置。

## 4. 从模型代码散落到统一抽取器

V4 已经使用 LoRA 和多种外部模型，但模型加载和调用分散在不同脚本中。

V5 把节点抽取抽象为：

- `QwenMaxExtractor`；
- `LocalFinetunedExtractor`；
- `DualExtractor`；
- `NodeExtractorFactory`。

这使本地模型、API 和双通道至少拥有统一入口，之后，本地抽取器从进程内加载改为 OpenAI-compatible 的 vLLM 远程服务，而 Pipeline 的主要职责不需要整体重写。

## 5. 从一条长脚本链变成阶段化流水线

V5 将任务拆为：

```text
pre        参考语料与参考图
post       讨论处理与参与者证据
scoring    十六维评分
summary    六大能力汇总
report     最终报告
```

`run_two_stage.py` 仍然使用子进程，这说明 V5 没有否定 V4 的进程隔离思路，而是减少了需要直接编排的脚本数量，并把细粒度步骤收回模块内部。

这是一次值得保留的折中：外层继续使用适合长任务的进程边界，内层使用类、Registry 和共享缓存提高复用性。

# 四、V4 到 V5 的差异总结

| 维度 | V4  | V5  |
|---|---|---|
| 代码组织 | 根目录编号脚本 | `src` 下按领域分包 |
| 流程表达 | 文件名编号 + `SCRIPTS` 顺序列表 | `PipelineRunner` + 阶段化入口 |
| 评分器 | 每个维度一个完整程序 | `BaseScorer` + `SCORERS` Registry |
| 配置 | 编排器生成大型路径字典 | `configs/settings.py` + 路径对象 |
| 数据访问 | 各脚本自行读写文件 | `DataManager` 集中读取和缓存 |
| 模型调用 | LoRA、DeepSeek、智谱等散落在脚本 | API/本地/双通道统一抽取接口 |
| 并发 | 多个脚本各自创建线程池 | Pipeline、Scoring、Reporting 分层并发 |
| 报告 | 独立 `summary.py` 和报告脚本 | `reporting` 包统一管理 |

V5 把“按执行步骤组织代码”改成了“按变化原因组织代码”：

一个评分算法变化，只需要进入某个 scorer；模型部署变化，只需要进入 extractor；报告格式变化，只需要进入 reporting。不同变化不再都穿过同一个根目录脚本链。

# 五、写在最后

从 V1 到 V5，这个项目从头到尾只有我一个人在写、在改、在推翻重来。踩过的坑、绕过的弯路，如果不趁现在记下来，下一次复盘大概又要对着旧代码考古。

所以我想以 V5 为基线，开一个系列，把这套系统从头到尾写清楚，做一次系统的复盘与温故知新——把做过的事讲明白，本身就是在重新学习。

即便如此，有两件事我会贯穿整个系列反复盯：

一是速度。当前最大的瓶颈就是耗时：两万字以上的文稿，跑完一次评估至少要一个小时。这个时间成本不压下来，客户就既难满意、也难下定采用的决心，它始终是落地前的一块绊脚石。

二是指标本身是否站得住。十六个维度现在都能算出分，但"算得出"不等于"算得对"。比如：
- 节点数量是否受到发言时长影响；
- 语义距离是否真的代表创造性或跑题；
- 中心性是否会随着小组人数变化；
- 组内 min-max 缩放是否允许跨组比较；
- LLM 摘要是否与底层证据一致。

这些问题不会因为代码更面向对象就自动消失。把它们一个个验证清楚，是这个系列想逼自己留下的功课，也是我从一个"能把系统跑起来的人"，往"能判断系统对不对、也能把来龙去脉讲清楚的人"再走一步的路。
