# 一个 Two-stage 的脚本，我为什么要分四段

这个项目的总入口叫 `run_two_stage.py`，文件开头也只写了两个阶段：

```text
Stage A (pre):  build reference nodes/arcs
Stage B (post): process discussions + scoring
```

但实际任务明显不止两种职责。参考材料先被处理成参考图（题目中可能涉及的信息点、决策和目标及其关系），讨论文本再被映射成每位参与者的个人证据（他实际提到了哪些概念、验证了哪些关系），十六个评分器随后计算分数，最后还有能力汇总、标签提炼和 PDF 报告。

为什么不把它们塞进一个 `main()` 从头跑到底？

因为 AI 长任务真正难处理的，往往不是"怎么再调用一次模型"，而是下面这些问题：

- 哪些结果可以被同一批参与者复用；
- 某一步失败时，应该从哪里重新开始；
- GPU、外部 API、CPU 和 PDF 渲染分别由谁负责；
- 一个空结果究竟代表"没有发现"，还是"这一段没有处理成功"；
- 最终报告能否回到上游中间产物进行检查。

**阶段化的价值，是先给这些问题划出边界。**

# 四段的数据流

在进入具体阶段之前，先看一张全局数据流图，标注的是每段产物的生命周期特征：

```text
参考语料 ──[pre]──→ 参考节点·参考关系 ──可跨参与者复用──┐
小组转写 ──[post]──→ 参与者节点·验证关系 ──按人归属─────┤
                                ┌─────────────────┘
                                ↓
                          [score]→ 16维分数 ──按维度独立──┐
                                                       │
                          [report]→ 能力汇总·标签·PDF ──业务交付
```

- **可跨参与者复用**：同一张参考图服务所有参与者，理论上只需构建一次
- **按人归属**：每位参与者的证据独立生成、独立保存，不能混用
- **按维度独立**：十六个评分器可以并发执行，单个维度失败不阻断其他维度
- **业务交付**：报告是给人看的最终产物，修改报告模板不需要重新跑模型

# 统一任务坐标

总入口要求 `project_id` 和 `group_id`，再通过 `reload_settings()` 生成本次任务的输入、输出和评分目录。调用者也可以手动传入路径覆盖默认值。

简化后的目录关系是：

```text
任务目录/
├── input/
│   ├── corpus/          # 参考材料
│   └── group_corpus/    # 小组讨论转写
└── output/
    ├── nodes/
    ├── arcs/
    ├── processed/
    ├── scores/
    └── result/
```

这一步看起来只是拼路径，实际上定义了四个阶段共享的任务坐标。`pre` 写入的参考节点，必须能被 `post` 找到；`post` 生成的参与者数据，必须能被评分器按同一个 speaker ID 读取；报告阶段又必须知道十六个评分目录属于哪一个项目和小组。

如果这些路径由每个 Agent 自行推断，文件系统就会变成隐式的全局变量。使用统一配置降低了这种风险。每个阶段通过读写约定好的文件格式来传递数据——比如 `pre` 写出 JSON 格式的参考节点，`post` 读取这些 JSON 并写出参与者节点的 JSON——文件本身就是阶段之间的协议。

# 总入口是一条显式命令链

`run_two_stage.py` 没有把 Pipeline 和 Scoring 全部导入同一进程，而是用子进程运行两个命令行入口，再在主进程调用 Reporting：

```python
run([
    "python", pipeline_script,
    "--phase", "pre",
    "--output_dir", str(data_dir),
])

run([
    "python", pipeline_script,
    "--phase", "post",
    "--output_dir", str(data_dir),
])

run([
    "python", scoring_script,
    "--data_dir", str(data_dir),
    "--output_dir", str(score_dir),
])

generate_summary(...)
generate_refined_summary(...)
generate_report_detail(...)
```

真实调用还会传入 provider、extractor、chunk size 和 worker 数量，这里只保留阶段结构。

为什么用子进程而不是直接调用函数？因为 Pipeline 和 Scoring 都会长时间占用 GPU、启动本地模型和批量调用外部 API。把它们放在独立进程里有三个好处：

* 如果某一段因为模型 OOM 或 API 异常崩溃，不会污染已经跑完的上游产物；
* 可以分别设置内存限制和环境变量；
* 进程退出后 GPU 资源会被操作系统自动回收。

缺点是阶段之间只能通过文件传递状态，不能在进程内共享模型缓存。

# pre：先建立公共问题空间

`pre` 只需要参考材料，不需要参与者讨论。Pipeline 读取参考语料后，依次做这几步：

1. **文本润滑**：对参考材料做口语化修正和噪声清理，让后续抽取更稳定
2. **分块**：把长文本切成适合模型处理的段落
3. **节点抽取**：让 LLM 从每个块中识别出信息点、决策项和目标
4. **去重与质检**：合并同名节点，剔除抽取质量不合格的结果
5. **分类**：给每个节点标上类型（信息节点、决策节点、效用节点）和抽象层级
6. **关系抽取**：在节点之间生成三类参考关系（因果、信息支持、证据更新）

核心输出包括：

```text
output/nodes/node_refanswer.json
output/arcs/caus_arcs.json
output/arcs/info_arcs.json
output/arcs/update_arcs.json
```

这四类文件共同定义了本场评估的参考概念空间：有哪些信息、决策和目标节点，哪些节点之间存在因果、信息支持或证据更新关系。

为什么把它单独放在 `pre`？因为它的生命周期与参与者不同。

同一个题目和参考材料可能对应一组参与者。参考图原则上只需要构建一次，之后每位参与者都在同一张图上匹配证据。如果为每个人重新生成参考图，不仅重复消耗模型调用，还可能让不同人的评分基准发生漂移——同一场讨论的不同参与者应该对着同一份"参考答案"来评分，否则分数之间的差异就不再反映能力差异，而是评分基准的差异。

因此，`pre` 的架构意图不是"先做一点预处理"，而是先冻结一份可复用的公共问题空间。

# post：把一场讨论变成每个人的局部证据

`post` 处理的是另一种生命周期的数据：小组讨论已经发生，现在要把多人转写拆到每位参与者，并将个人表达映射回 `pre` 生成的参考空间。

它主要执行：

```text
小组讨论转写
  -> 按说话人拆分
  -> 润滑参与者文本（口语修正与噪声清理）
  -> 分批判断参考节点是否被提到（把全部参考节点每 20 个一组交给 LLM，问"这个人讨论过这个概念吗"）
  -> 保存个人节点
  -> 从参考图筛选候选关系（只保留头尾都在该参与者命中节点集合中的参考边）
  -> 回到个人原文验证关系（把候选边交给 ArcVerifyAgent，让它逐条回到发言原文确认"这个人确实把 A 和 B 连在一起说了吗"）
  -> 生成评分器需要的派生文件
```

后两步是 `post` 的核心逻辑，也是它和"自由抽取"的关键区别：先约束候选范围（节点和关系都来自参考图），再回到原文验证（不是模型凭空判断，而是检查发言文本是否真的支持这条关系）。

代表性输出包括：

```text
output/processed/split_corpus/
output/processed/smoothed_corpus/
output/processed/node_output/<speaker>.json        # 每位参与者命中的参考节点
output/processed/arc_output/<speaker>.json          # 每位参与者经验证的关系
output/processed/for_reasoning/<speaker>.txt        # 推理模式数据（用于思维激发评分）
output/processed/chance_node/<speaker>.txt          # 信息节点文本（用于信息收集评分）
output/processed/decision_node/<speaker>.txt        # 决策节点文本（用于目标拆解评分）
```

`post` 依赖 `pre`，它回答的是："在共同标准已经确定之后，这位参与者贡献了哪些证据？"

下面用一个结构演示说明两者的差别：

| 数据 | 示例内容 | 所属阶段 |
|---|---|---|
| 参考材料 | 题目背景、可选方案、目标和约束 | `pre` |
| 参考节点（题目中可以讨论的概念） | 客户到店量下降、调整服务时段、提升经营效益 | `pre` |
| 小组转写 | 三位参与者围绕方案进行讨论 | `post` |
| 参与者节点（这个人实际讨论了什么） | 到店量下降、调整服务时段 | `post` |
| 参与者节点（这个人实际讨论了什么） | 提升经营效益 | `post` |

参考节点描述"这场题里可以讨论什么"，参与者节点描述"这个人实际讨论了什么"。将两者拆开，才能让不同参与者共享标准，同时保留个人证据归属。

# score：证据准备完成后，16 个能力维度各自计算

评分阶段不再负责生成参考图，也不再拆分参与者文本。它通过 `DataManager`（一个集中读取和缓存 Pipeline 产物的共享数据层）读取上游数据，并从 `Registry`（一个字典，把评分维度 ID 映射到对应的评分类）中取得需要运行的评分器。

十六个评分维度分属六大能力：

| 能力 | 子维度 | 主要评分方法 |
|---|---|---|
| 思维能力 | 逻辑思维、抽象归纳 | 参考图关系模式 |
| 目标与方案 | 目标拆解、方案影响预判 | 参考图关系模式 |
| 信息与系统 | 信息收集、系统思维 | 节点类型与数量 |
| 创造与聚焦 | 创造性、聚焦性 | Embedding 距离 |
| 互动与协作 | 思维激发、时间把握、开放度、冲突消解、协作中心性 | Hawkes 点过程、规则、LLM、图计算 |
| 方案可行性 | 方案可行性与最终评语 | 结构化数据 + LLM |

评分入口的核心结构是：

```python
# ScoringPaths 封装了所有上游目录的路径，让评分器不需要自己拼路径
paths = ScoringPaths(base_dir=Path(args.data_dir))

# run_all_scorers 从 Registry 取得评分器列表，按参数选择全部或部分维度
results, timings = run_all_scorers(
    paths=paths,
    scorer_ids=args.scorers,
    generate_summary=not args.no_summaries,
    generate_plot=not args.no_plots,
    max_workers=args.workers,
    return_timings=True,
)

save_results(results, Path(args.output_dir))
```

每个评分器的结果被保存到独立目录：

```text
scores/<scorer_id>/scores.json
scores/<scorer_id>/details.json
scores/<scorer_id>/summaries/<speaker>.txt
```

把 `score` 独立出来，有三个直接收益。

第一，评分算法可以在不重新抽取节点和关系的情况下迭代。修改某个图模式、Embedding 距离或缩放公式时，理论上可以复用上游数据。

第二，十六个评分器可以并发执行，并共享 `DataManager` 的惰性缓存，不需要每个维度都重新理解完整目录结构。

第三，单个评分维度可以通过 Registry 选择性运行，便于定位问题和做局部实验。

# report：分数不是交付物，报告才是

评分目录解决的是机器可读结果，业务最终需要的是每位参与者能够被阅读和交付的报告。当前外层入口把 Reporting 又拆成三个内部步骤：

```text
summary
  -> 汇总 16 维分数
  -> 计算六大能力
  -> 生成雷达图和能力总结

refined summary
  -> 生成个人标签
  -> 把各维度总结转成优势和不足要点

report detail
  -> 聚合分数、图表、标签和要点
  -> 生成报告明细与 PDF
  -> 更新报告状态
```

最终报告包含每位参与者的能力雷达图、维度得分、优势与不足要点、个人标签和详细评语。它面向的读者是需要做综合判断的面试官或评估负责人。

本文把它们合并称为 `report`，是因为三者共同负责"从计算结果到业务交付"，并不意味着代码里只有一个 report 函数。

# 四段拆分带来了什么

## 1. 数据复用边界更清楚

`pre` 面向参考材料，`post` 面向参与者讨论，`score` 面向算法结果，`report` 面向交付物。数据生命周期不同，复用方式也不同。

如果所有步骤混在一起，修改报告模板也可能迫使参考图重建；拆开后，至少可以看见哪些工作本来不应该重复。

## 2. 失败域可以分层讨论

外层 `run()` 使用：

```python
subprocess.run([str(x) for x in cmd], check=True)
```

只看 `subprocess.run(check=True)`，很容易得出"任何错误都会阻断任务"的结论。当前实现并不是这样。

Pipeline 内部的部分批次失败会被捕获并记录；节点判断 API 异常时可能直接返回空列表。Scoring 会分别捕获单个 scorer 的异常，把该维度结果记为空，然后继续运行其他评分器。Summary 和 Report 也会按参与者捕获异常，避免一个人的失败拖垮全组。

因此，当前同时存在两种错误语义：

```text
阶段级失败
  -> 子进程非零退出
  -> 外层停止后续阶段

阶段内局部失败
  -> 记录日志或返回空结果
  -> 阶段仍可能以 0 退出
  -> 外层继续执行
```

也就是说，系统会尽量跑完所有能跑的部分，而不是遇到第一个错误就全盘放弃。但缺点是，"外层成功"不必然等于"每个参与者、每个维度、每份 PDF 都完整成功"。

## 3. 资源模型可以分别设计

四段的主要瓶颈并不相同：

| 阶段 | 主要资源或外部依赖 |
|---|---|
| `pre` | 节点抽取模型、LLM API、关系组合任务 |
| `post` | 多参与者 × 多节点批次的 LLM 调用 |
| `score` | CPU、Embedding 模型、LLM、Numba 和绘图 |
| `report` | LLM 总结、文件聚合、图表与 PDF 渲染 |

只有先拆开阶段，才有可能分别设置 worker 数、模型生命周期、超时和资源预算。

## 4. 可观测性有了天然切点

Pipeline、Scoring、Summary、Refined Summary 和 Reporting 分别写入 timing JSON，外层再合并成：

```text
output/result/timings_merged.json
```

当前 timing 还不能替代完整监控，但它至少允许回答"时间花在 Pipeline、评分还是报告"，而不是只得到一个端到端总耗时。

## 5. 阶段可以拥有不同的变化理由

节点抽取模型变化，应主要影响 Pipeline；新增评分维度，应主要影响 Scoring；报告版式变化，应主要影响 Reporting。

这正是模块边界的含义：不是文件放在不同目录，而是每一类变化有更明确的代码所有者。

# 小结

并不是所有 AI 应用都应该照搬 `pre/post/score/report`。真正可迁移的是拆分依据。

设计一个长任务时，可以依次问五个问题：

1. **哪些输入会被多次复用？** → 参考图只需构建一次，多位参与者共享同一份参考节点和关系
2. **哪些步骤拥有不同的资源瓶颈？** → Pipeline 瓶颈在 LLM API，Scoring 瓶颈在 CPU 和 Embedding，Reporting 瓶颈在 PDF 渲染
3. **哪些结果需要被人检查、重放或单独重算？** → 报告模板改了只需要重跑 report，某个评分公式改了只需要重跑对应维度
4. **某一步失败后，哪些结果仍然有效？** → pre 成功后 post 部分失败，参考图仍然可用；某个评分维度失败不影响其他十五个维度
5. **哪一类变化应该由哪一个模块负责？** → 抽取模型变化归 Pipeline，评分公式变化归 Scorer，报告版式变化归 Reporting

如果两个步骤的答案明显不同，它们通常值得成为不同阶段。

对当前 V5 版本来说：

```text
pre       固定公共问题空间
post      归属个人讨论证据
score     将证据转换为能力指标
report    将指标转换为业务交付
```

这四段让系统比"一次 Prompt 直接出报告"更容易复用、检查和演进。当前 V5 的运行形态，是一条由 CLI 启动、按固定顺序编排、通过文件传递中间结果的批处理 Workflow。它适合整场讨论结束后的集中评估，但尚未具备统一的任务 manifest、通用 checkpoint 和完整的部分成功协议。

阶段化不是完成可靠性治理，而是让可靠性问题终于有了可以落脚的位置。
